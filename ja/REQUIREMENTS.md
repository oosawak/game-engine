# 要件定義

## 概要

この文書は、ゲームエンジン Core MVP の初期要件を定義する。

## 機能要件

### エンジンコア

- エンジンは安全に初期化と終了ができること
- エンジンは決定的な更新ループを実行できること
- エンジンは Delta Time を計算して公開できること
- エンジンは目標 FPS または更新周期を設定できること

### シーン管理

- 複数シーンを扱えること
- シーンを登録できること
- アクティブシーンを切り替えられること
- アクティブシーンに update と render を伝播できること

### GameObject 管理

- シーンが複数の GameObject を管理できること
- GameObject に一意の識別子があること
- GameObject が親子関係を持てること
- GameObject を安全に削除・破棄できること

### Component システム

- GameObject に Component を追加できること
- 型指定で Component を取得できること
- Component は `start`、`update`、`destroy` のライフサイクルを持つこと
- `start` は有効化ごとに 1 回だけ実行されること
- `Transform` は全 GameObject に必須であること

### Transform と数学

- 座標計算に必要な基本数学型を提供できること
- Transform は position、rotation、scale を扱えること
- Transform は基本的な移動と回転操作を持つこと

### テスト

- コアのライフサイクル動作を単体テストで検証できること
- シーン切り替えをテストできること
- GameObject の破棄動作をテストできること

## 非機能要件

- アーキテクチャはモジュール分割されていること
- コアは深いフレームワーク知識がなくても理解できること
- ブラウザ実行に適していること
- 将来の WASM / Rust 統合に対応しやすいこと
- 早すぎる最適化より保守性を優先すること

## 制約

- 初期 MVP は完全なレンダラーに依存しないこと
- 初期 MVP は物理エンジン統合を必要としないこと
- 初期 MVP はネットワーク統合を必要としないこと
- 初期 MVP は ECS への移行を先回りしすぎないこと

## 受け入れ条件

- Engine の start / stop が動作する
- シーンの登録と切り替えが動作する
- GameObject の生成、追加、更新、破棄が動作する
- Component の実行順序が正しい
- GameObject に Transform が必ず存在する
- 基本テストが通る

## 将来の拡張点

- 描画バックエンド
- 入力バックエンド
- リソース読み込み
- 物理統合
- ネットワークプロトコル
- エディタ統合
- ECS 移行
